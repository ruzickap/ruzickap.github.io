# AGENTS.md

Personal Jekyll blog (`ruzickap.github.io`) using the **Chirpy** theme.
Content lives in `_posts/<year>/`. The twist: many posts are also
**executable integration tests** that spin up/tear down real Amazon EKS
clusters, driven by the code-fence language inside the Markdown.

## Critical: code-fence language is load-bearing

In `_posts/**`, the fence language of shell blocks controls execution. Do not
change it casually, and pick the right one when adding blocks (`scripts/mise-create-delete-posts.sh`,
`.github/workflows/mega-linter.yml`):

- ` ```bash ` — extracted and **run on `create`** (cluster setup steps).
- ` ```sh ` — extracted and **run on `delete`** (cleanup/teardown steps).
- ` ```shell ` — **display-only**, never executed (creds, illustrative output).
  Use this for anything that must not run, e.g. example `export AWS_ACCESS_KEY_ID=...`.

All three (`bash`/`shell`/`sh`) are still extracted into one script by MegaLinter
for `shellcheck` syntax validation, so every block must be valid shell.

## Post testing (`mise` + `mq`)

- Tasks are defined in `mise.toml` as `create:<post>` / `delete:<post>`. Post
  names with `|` are dependency chains, e.g.
  `create:2025-02-01-eks-auto-cert-manager-velero|2024-12-14-secure-cheap-amazon-eks-auto`
  layers that post on top of the base cluster post.
- `mq` (a Markdown query tool) extracts the fenced blocks; `mise run <task>`
  executes them via `scripts/mise-create-delete-posts.sh`. Requires real AWS
  credentials — the script aborts if `aws sts get-caller-identity` fails.
- Run everything (expensive, real infra): `mise run "create-delete:posts:all"`.
- CI entry point is the manual `post_tests.yml` workflow (workflow_dispatch only).
- When adding a new tested post, add matching `create:`/`delete:` tasks to
  `mise.toml` **and** an option in `post_tests.yml`.

## Connecting to the EKS / Kubernetes cluster

- To get `kubectl` access to the live EKS cluster, run **`eval "$(mise run a)"`**
  (`a` is the alias for the `eks-access` task). It sources the base post's env
  setup (`AWS_REGION`, `CLUSTER_NAME`, …) and runs `aws eks update-kubeconfig`
  into a throwaway `KUBECONFIG`, so `kubectl` works in the current shell
  afterwards.
- This is a **one-time, per-session** action — run it **once** per shell
  session, then reuse the same session for all `kubectl` commands. Do not
  re-run it before every command, and do not hand-roll
  `aws eks update-kubeconfig` yourself.
- It needs working AWS credentials (`aws sts get-caller-identity` must succeed).

## Build & lint locally

- Build the site: `bundle exec jekyll build --destination public`
  (Ruby 3.4.x, gems via `bundle install`).
- HTML check (matches CI): `bundle exec htmlproofer public --disable-external`.
- Full lint suite is **MegaLinter**, not standalone tools:
  `mega-linter-runner --remove-container --container-name="mega-linter" --env VALIDATE_ALL_CODEBASE=true`.
  Config is `.mega-linter.yml`.

## Repo-specific conventions

- Markdown is linted with **`rumdl`** (not markdownlint); config `.rumdl.toml`.
  `MD013` line length ignores code blocks, but prose should wrap at 80.
- Shell formatting via `shfmt`: `--case-indent --indent 2 --space-redirects`.
  `shellcheck` excludes `SC2317`.
- `CHANGELOG.md` is auto-generated by release-please — never edit it; it is
  excluded from all linters.
- `_tabs/projects.md` is generated by `scripts/generate-projects.sh`
  (scheduled `generate-projects.yml`) — edit the script, not the file.
- `_config.yml` placeholders are intentional: `url`, cloudflare analytics `id`,
  etc. are injected at deploy time by `gh-pages-build.yml`. Don't "fill them in".
- Tooling is pinned in `mise.toml` (aws, eksctl, kubectl, opentofu, helm, ...);
  prefer `mise`-provided versions over system installs.
- `fnox.toml` resolves AWS-stored secrets locally via mise; CI deletes it and
  uses GitHub secrets instead (see `post_tests.yml`).

## Git / PR workflow

- Conventional Commits enforced by `commit-check` (subject ≤ 72 chars,
  imperative, lowercase, no trailing period). PR titles validated by
  `semantic-pull-request`.
- Branches: Conventional Branch format `<type>/<description>`
  (`feat/`, `fix/`, `chore/`, `release/v1.2.0`), lowercase + hyphens only.
- **Open PRs as drafts.** Pin GitHub Actions to full SHAs; keep
  `permissions: read-all` unless a job needs more. Validate workflow edits with
  `actionlint`.

## Layout

- `_posts/<year>/` — blog posts (content + executable tests).
- `_tabs/` — site pages (`about`, `archives`, generated `projects`, etc.).
- `_data/`, `_plugins/` (`posts-lastmod-hook.rb`), `assets/` — Jekyll internals.
- `scripts/` — post test runner and projects generator.
- `.github/workflows/` — CI (mega-linter, gh-pages-build, post_tests, ...).
